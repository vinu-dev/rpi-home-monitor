# Hardware Requirements

Status: Draft prepared to support expert regulatory review.

Hardware exists and is implied by the repository. Hardware requirements are
therefore applicable.

| ID | Statement | Rationale | Source | Parent links | Risk/security links | Verification method | Linked tests | Status |
|---|---|---|---|---|---|---|---|---|
| HWR-001 | The server hardware shall be Raspberry Pi 4B or documented equivalent with sufficient RAM for server, recording, and UI services. | Server capacity is central to recording reliability. | `docs/guides/hardware-setup.md` | SYS-011 | RISK-001, RISK-008 | Hardware smoke/manual review | TC-018 | Draft |
| HWR-002 | Camera hardware shall use Raspberry Pi Zero 2W or documented equivalent with supported camera module and hardware encoder constraints. | Camera capture limits drive safe mode selection. | `docs/guides/hardware-setup.md`, `sensor_info.py` | SYS-011 | RISK-001, RISK-007 | Unit guardrails and hardware smoke | TC-018 | Draft |
| HWR-003 | Server and camera nodes shall provide WiFi/Ethernet network interfaces sufficient for local streaming and management. | Network availability is required for live view, recording, pairing, and OTA. | `docs/guides/hardware-setup.md` | SYS-001, SYS-006 | RISK-005, SC-004 | Integration and smoke tests | TC-006, TC-018 | Draft |
| HWR-004 | Server storage shall include SD storage and support configured external USB storage for recordings. | Recording retention depends on writable storage. | `docs/history/baseline/requirements.md` | SYS-002, SYS-003 | RISK-003 | Storage tests, manual storage validation | TC-003, TC-018 | Draft |
| HWR-005 | Devices shall be powered by stable supplies appropriate for the Raspberry Pi models used. | Brownouts can cause data loss and false offline states. | `docs/guides/hardware-setup.md` | SYS-011 | RISK-008 | Manual hardware validation | TC-018 | Draft |
| HWR-006 | Server and camera images shall support hardware-mediated reset or reflash recovery rather than software backdoor reset. | Recovery must not undermine authentication. | ADR-0022 | SYS-008 | RISK-006, SC-006 | Factory reset tests/manual review | TC-015, TC-018 | Draft |
| HWR-007 | Camera overlays and sensor detection shall support the documented sensor set and avoid exposing unsupported encoder modes. | Prevents camera boot or stream failure after configuration. | ADR-0023 | SYS-006, SYS-011 | RISK-007 | Board/profile tests | TC-012, TC-018 | Draft |
| HWR-008 | System time shall be synchronized through the local server/camera time-sync design. | Correct timestamps are required for clips, events, audit logs, and updates. | ADR-0019 | SYS-002, SYS-010 | RISK-005 | Version/time design tests and review | TC-017, TC-018 | Draft |
| HWR-009 | Camera sensor, CSI cable, lens, and mount shall match the supported sensor/capability matrix for the selected image. | Unsupported camera hardware can silently eliminate monitoring. | `docs/guides/hardware-setup.md`, ADR-0023 | SYS-011, SYS-006 | RISK-007, RISK-022 | Sensor capability and hardware smoke tests | TC-012, TC-047 | Draft |
| HWR-010 | Camera hardware shall provide visible or documented local indicators and recovery inputs for setup, connecting, connected, and error states. | Headless installation requires operator feedback and recovery affordances. | ADR-0013 | SYS-025 | RISK-022 | LED/status tests and manual inspection | TC-039, TC-047 | Draft |
| HWR-011 | Removable storage used for recordings shall be operator-controlled media and support safe mount, format, eject, and failure detection workflows. | Uncontrolled storage can cause data loss or accidental formatting of unrelated media. | `docs/guides/hardware-setup.md` | SYS-016, SYS-003 | RISK-003, RISK-013 | USB/storage tests and manual validation | TC-024, TC-025 | Draft |
| HWR-012 | LAN equipment shall support the documented local services, mDNS discovery, HTTPS/RTSPS/WebRTC transport, and optional local time synchronization. | Local-first operation depends on predictable network behavior. | ADR-0005, ADR-0019 | SYS-015, SYS-019, SYS-029 | RISK-012, RISK-017 | Network/live validation | TC-023, TC-028, TC-047 | Draft |
| HWR-013 | Server and camera deployments shall remain within documented thermal, power, and ventilation assumptions for the Raspberry Pi models used. | Thermal throttling or brownouts can degrade monitoring without obvious software failure. | `docs/guides/hardware-setup.md` | SYS-011, SYS-025, SYS-030 | RISK-008, RISK-022 | Hardware smoke/manual validation | TC-035, TC-047 | Draft |
| HWR-014 | Server and camera SD/media shall support power-loss recovery expectations for configuration, logs, OTA state, and recordings. | Sudden power loss can corrupt state needed for recovery and evidence. | ADR-0002, ADR-0008 | SYS-021, SYS-023 | RISK-004, RISK-012, RISK-019 | OTA/config/storage validation | TC-013, TC-033, TC-047 | Draft |
| HWR-015 | Production images shall include documented firewall and service exposure settings for server and camera roles. | Hardware roles expose different services and should not share debug surfaces. | firewall configs, Yocto image configs | SYS-029, SYS-030 | RISK-017, RISK-018 | Production-profile and network checks | TC-044, TC-047 | Draft |
| HWR-016 | Deployment shall document physical access assumptions for SD cards, USB media, camera placement, reset inputs, and exposed ports. | Physical access can bypass or weaken software controls. | ADR-0010, ADR-0022 | SYS-008, SYS-028, SYS-030 | RISK-006, RISK-019 | Security review/manual validation | TC-015, TC-032, TC-047 | Draft |
| HWR-017 | Supported camera count, bitrate, resolution, FPS, and storage-retention limits shall be defined for each hardware profile before production use. | Capacity limits determine whether monitoring and storage requirements are realistic. | ADR-0015, ADR-0023 | SYS-001, SYS-002, SYS-011 | RISK-001, RISK-007, RISK-022 | Performance and hardware smoke validation | TC-001, TC-018, TC-047 | Draft |
| HWR-018 | Build, signing, and release hosts shall keep development credentials, production signing material, and generated artifacts separated by documented workflow. | Build host handling affects production artifact integrity. | `docs/guides/ota-key-management.md`, release workflows | SYS-023, SYS-024, SYS-028 | RISK-018, RISK-019 | Release/security review | TC-043, TC-045 | Draft |
